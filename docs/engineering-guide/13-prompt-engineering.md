# 提示词设计：这个系统的灵魂

坦白说，Agent 架构大家都能抄——ReAct 循环、工具注册、向量检索，这些模式已经很成熟了。但这个系统之所以能做好深度研究，核心在于**提示词怎么设计**。这篇笔记不讲空泛的 prompt engineering 理论，而是从 agent_system.txt 的实际文本出发，分析每条关键句子的设计意图。

## 提示词的分层体系

提示词分了四层，各有各的职责：

```
Layer 1: agent_system.txt（内核层）
  不变的研究哲学、核心原则、工具优先级
  所有 session 共享，编译后就冻结

Layer 2: SOUL.md + AGENT.md + MEMORY.md（会话层）
  每个 session 独立的身份设定和行为指南
  在 session 创建时生成，可被任务动态更新

Layer 3: Skills（领域/任务层）
  特定类型任务的扩展指令
  按需加载，用完即弃

Layer 4: <system-reminder>（动态注入）
  时间戳、文件变更通知、todo 提醒
  追加在消息尾部，不影响 KV cache
```

为什么分这么多层？因为 KV cache。system prompt 是 cache key 的前缀，每次变化都导致缓存失效。所以不变的放第一层，半变的放第二层，动态的放第四层，并且严格保证已有的 cache prefix 不被破坏。

## agent_system.txt 逐节拆解

这个文件在 `agent_os/prompts/agent_system.txt`，大约 90 行，用 XML 标签分成几个节。下面逐节讲为什么这么写。

### `<角色>`（3 行）

```
你是运行在 AgentOS 上的深度研究 AI 助手。你可以搜索、阅读、推理、验证和综合信息。
你的工作不是浏览更多页面，而是构建一个可辩护的答案模型。
```

最有力量的是最后一句："不是浏览更多页面，而是构建一个可辩护的答案模型"。这句话直接定义了 Agent 的价值导向——它不是一个信息收集器，而是一个论证构建器。所有搜索、阅读的行为最后都要服务于一个可辩护的答案。这也是 Agent 停止工作的信号：当答案可辩护时就停止，而不是搜到没有新东西为止。

### `<核心原则>`（10 行）

这是整篇提示词最重要的部分，定义了研究的哲学基础。核心就一句话：

**"研究是以检索为验证的约束驱动推理。"**

把研究拆解成：推理（hypothesis）→ 检索（verify）→ 修正（refine）的循环。不是"搜索→阅读→总结"的机械流程。

然后给出了非常具体的"怎么做"和"别怎么做"：

| 原文 | 意图 |
|------|------|
| "使用对应的检索工具验证你的主张" | 不要靠已有知识，去查 |
| "我觉得下一步可能的动作或者候选集应该是什么" | 假设驱动的搜索，不是泛泛搜关键词 |
| "把事实和假设区分开来，不能把推理过程的假设当成具体事实" | 元认知 |
| "检索停滞时可能的失败原因是错误的前提、薄弱的框架或缺失的候选集，而不仅仅是关键词不好" | 搜不到的时候换方向，不是换关键词继续搜 |

最后这句"不仅仅是关键词不好"非常关键。我观察到新手 Agent 经常会做的一件事就是：搜不到 → 换一组关键词 → 还搜不到 → 再换一组。无限循环。这句提示词给了 Agent 一个出路——停下来想想前提是不是错了。

### `<任务模式路由>`（8 行）

```
在研究之前确定 answer_contract.mode：
- short_answer：一个事实/实体/数字/年份/地点
- long_form_report：分析、比较、综合、结构化报告
- interactive_research：目标不够明确，先提澄清问题
```

Agent 在动手检索之前，必须先判断任务的模式。为什么要显式路由？因为两种模式对工具使用、检索深度、输出格式的要求完全不同，混合在同一个 prompt 中会导致两边都做不好。

比如 short_answer 只需要精确命中，不需要全面覆盖；long_form_report 需要源策略、覆盖映射、引用协议。如果告诉 Agent "既要全面又要精准"，它就会困惑。

路由是通过 skill_use 加载专门的 skill 提示词来实现的，而不是在 system prompt 里塞两套指令。这保持了 system prompt 的精简。

### `<通用研究循环>`（8 行）

```
1.分解问题
2.找到最小切入点（可以直接使用结构化工具或者权威信源快速验证），
  绝对不能找一个非常发散的点作为切入
3.调研验证（思考，检索，复盘，继续检索，复盘）
4.形成答案（写作，回看证据，补充缺漏，排除幻觉）
时时回顾，循序向前
```

"最小切入点"这个说法是我比较满意的设计。Agent 被要求不要从大问题开始搜，而是找一个可以快速验证的小入口。比如要回答"XX 物质是什么"，不是搜"XX 物质"，而是先搜 "XX 年份 意外 发现 物质"。前者太泛，后者直击要害。

### `<推理策略>`（2 行）

```
选对过滤锚点比多搜几次更重要。
```

这是唯一用了 `**` 加粗的句子。故意的。整个 prompt 里只有这一句加粗，模型能注意到。

这句话支撑了整套搜索方法论：先确定领域 → 找到权威源 → 用 site: 深入。不是广撒网。

### `<运行时控制>`（5 行）

```
由于你后续是看不到自己的思维过程的，这及其容易导致思维冗余和幻觉以及线索偏移，
所以你必须及时的将你的思考反映到纸上，防止后续跑偏。
```

这里解释了为什么需要 research_state 工具。Agent 在推理过程中产生的 reasoning tokens，在下一轮调用时是不进上下文的（模型架构限制）。所以必须把思考过程写到文件里（research_state），下一轮才能读到。不写就会跑偏，而且自己没有察觉。

research_state 的自由格式也是有意的——不给模板，让模型自己选择最佳的组织方式。实战中模型会自然地形成几种模式：状态快照、假设生成、回溯复盘、收敛检查。

### 其他几个节

`<工具使用>` 定义了工具优先级：专用工具优先于通用搜索（比如法律问题先 law_retrieve 而不是 web_search），检索结果自动归档到 raw_search/。

`<验证>` 要求关键事实有两个以上独立来源，并且显式标注置信度。

## 运行中遇到的失败模式

基于实际使用，整理了一些最常见的失败场景，这些也是在提示词迭代中逐步加入防护的：

**"搜了再说"**：Agent 收到问题后不做任何分析，直接搜索原始问题。连续 10+ 次搜索不回写不反思。应对：`blind_search_guardrail_rounds` 配置项，强制在第 9 轮搜索后停下来写 research_state。

**幻觉性搜索**：搜索自己脑中猜想的关键词组合，实际命中率为 0。应对：提示词中说明检索停滞时应反思前提而非关键词。

**假设变成事实**：第一轮推理说"可能和 A 有关"，第二轮就变成了"因为 A 和 B 有关联，所以……"。应对：前面说的"把事实和假设区分开来"。

**发散切入**：找一个非常复杂的切口，而不是从最简单的验证开始。应对："绝对不能找一个非常发散的点作为切入"。

**过度依赖通用搜索**：不用专用工具（law_retrieve, arxiv_search），而是用 web_search 搜专业内容。应对：通过 skill 的 allowed-tools 限制工具集。

## 和其他系统对比

| 维度 | AgentOS | 常见的 LangChain Agent |
|------|---------|----------------------|
| system prompt | 固定 XML 结构，一次编译终身使用 | 每次请求动态组装 |
| 任务路由 | 显式路由（short_answer vs long_form） | 无，混在一起 |
| 反思机制 | 内置于循环 + research_state 外化 | 无 / 外部 callback |
| 搜索策略 | 层次化（先锁定源，再 site: 深入） | 简单 tool loop |
| 错误恢复 | 两级内容过滤恢复 + guardrails | 简单重试 |
| KV cache | 首席考量 | 通常不考虑 |

## 一点经验

这个提示词不是一次写成的。经历了大约 6 个版本的迭代：

1. 最开始是法律专用，只能处理法律检索
2. 通用化后发现容易"搜了再说"
3. 加入"约束驱动推理"哲学后行为显著改善
4. 加入模式路由解决 short_answer 和 long_form 冲突
5. 加入 research_state 解决看不到自己推理的问题
6. 加入 blind_search_guardrail 和内容过滤恢复让它更健壮

每次修改解决一个具体的失败模式。好的提示词不是写出来的，是迭代出来的。
