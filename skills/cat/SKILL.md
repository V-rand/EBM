# Critically Appraised Topic (CAT) — 临床证据快评

## 核心原则

CAT 是写给临床医生看的——像同事在咖啡机旁跟你解释一篇论文那样写，不是填表。

**不要写模板化报告。** 市面上 AI 报告的通病：PICO 表→检索策略→证据表→RoB 表→CBL，千篇一律、全是表、大量废话。CAT 要的是：结论开头→用段落解释为什么→需要细节才展开。问题简单就短，问题复杂才长。

## 写作质量——什么样的报告才像人写的

### 1. 结论在前，解释在后

第一段就是 Clinical Bottom Line。不需要"检索了 PubMed 找到 N 篇"——可以直接写结论。PICO 不需要单独做一个表——融入到第一段里。

**差的（模板化）：**
```
## PICO 框架
| P | I | C | O |
|---|---|---|---|
| T2DM合并CVD | SGLT-2i | 安慰剂 | MACE |

## 检索策略
数据库: PubMed, Cochrane
检索词: SGLT-2 inhibitor...

## 证据概要
| 研究 | 设计 | N | HR | GRADE |
...
```

**好的（人写的）：**
```
对于 T2DM 合并 ASCVD 的患者，SGLT-2 抑制剂可降低 MACE 风险约 14%
（HR 0.86, 95%CI 0.74-0.99），基于 EMPA-REG OUTCOME 试验。这意味着
每治疗 39 名患者 3 年可预防 1 例 MACE。

这个结论来自 FLAURA 试验——一项多中心双盲 RCT，将 556 名初治 EGFR
突变 NSCLC 患者随机分配到 osimertinib 或一代 TKI...

值得注意的是，VERTIS CV（ertugliflozin）在 MACE 上未达到显著差异
（HR 0.97），提示可能不是真正的 class effect。
```

注意区别：好的版本从结论开始，用自然段落解释，需要细节时才展开数据。没有"PICO 表"、"检索策略"这样的章节标题。

### 2. 可变长度——不是每个问题都要写满 200 行

| 问题复杂度 | 长度 | 内容 |
|-----------|------|------|
| 简单事实（"X 药的半衰期"） | 1 段 | 直接回答 + 来源 |
| 单一 PICO（"X 药在这个人群有效吗"） | 3-5 段 | CBL + 关键证据 + 局限 |
| 多药对比或亚组复杂的 | 10-15 段 | CBL + 逐药分析 + 证据链 + 对比表 |

如果只有一篇 landmark RCT 回答这个问题，就不要写 5 段——2 段就够了。不要为了"看起来完整"而堆内容。

### 3. 表格只在真正需要对比时用

表不是报告的主体——表是当你需要**对比多项研究的同一结局**时才用的工具。以下场景不需要表：
- 只有 1 篇主要研究 → 段落描述即可
- 效应量可以用一句话写完 → 不需要表
- 检索策略 → 完全不需要写在报告里（读者不关心你怎么搜的）

**需要表的时候：** 3 种以上药物头对头对比、亚组效应方向不一致、安全性 profile 对比。

### 4. 删掉废话和过程

以下内容永远不要出现在报告中：
- "我们检索了 PubMed 和 Cochrane Library" —— 读者不需要知道
- "经过 GRADE 评估" —— 直接写 GRADE 结果
- "本 CAT 具有一定的局限性" —— 直接说局限是什么
- "值得注意的是"、"综上所述"、"深入探讨" —— 全部删掉

### 5. 写你真正检索到的，不要填模板

如果你没搜到指南，就不要写"指南层：无"——直接跳过指南层，从你实际找到的证据开始写。如果你只找到 1 篇 SR 和 2 篇 RCT，就围绕这 3 篇写，不需要为"完整"而虚构层级结构。
